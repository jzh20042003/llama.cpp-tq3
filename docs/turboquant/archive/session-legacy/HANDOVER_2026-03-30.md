## Architecture

### TQ3_1S Block Format

```
[d0: fp16][d1: fp16][qs: 12 bytes] = 16 bytes per 32 elements (4.0 bpw)
```

- d0: scale for elements 0-15
- d1: scale for elements 16-31
- qs: 3-bit packed indices into 8 centroids
- Weights are WHT-rotated before quantization

### CUDA Kernel: MMVQ (Token Generation)

**File:** `ggml/src/ggml-cuda/vecdotq.cuh`
**Function:** `vec_dot_tq3_1s_q8_1`

- Activations are pre-rotated into WHT domain via `ggml_cuda_tq3_rotate_act`
- Quantized to q8_1 format
- Dot product computed directly against centroids with dual scale selection
- `VDR_TQ3_1S_Q8_1_MMVQ = 4` (4 groups of 8 elements)
- GPU/CPU PPL ratio: 1.0001 (verified correct)

### CUDA Kernel: MMQ (Prefill) — DISABLED

**File:** `ggml/src/ggml-cuda/mmq.cuh`
**Function:** `load_tiles_tq3_1s`

- Written and compiles, but produces NaN after ~4 chunks
- Root cause: q8_0 requant format stores one `d` per 32 elements, but TQ3_1S needs two
- Lane 0 stores d0, so groups 2-3 get wrong scale → error accumulates → NaN
- Currently disabled via `tq3_1s_mmq_ok` flag in `ggml-cuda.cu`
- Prefill falls back to cuBLAS (still fast: 221 tok/s)

### Dispatch Rules (ggml-cuda.cu)

```
TQ3_1S MMVQ:  enabled for TG (ne[1]==1), blocked for prefill (ne[1]>1)
TQ3_1S MMQ:   disabled (NaN bug)
TQ3_1S cuBLAS: used for prefill — dequant to fp32 + GEMM
Pre-rotation:  both TQ3_0 and TQ3_1S rotate activations before MMVQ
Contiguity:    TQ3_1S requires contiguous src0 for MMVQ direct path
```

### KV Cache

TQ3_0 KV cache quantization works with TQ3_1S weights (expert's existing code):
- `-ctk tq3_0 -ctv tq3_0` → 4.6x compression vs fp16
- Enables 8192 context on 16GB GPU (112 MB KV vs ~1 GB fp16)
- No new code needed — already in the codebase

## Known Issues

### 1. MMQ NaN (Priority: Medium)
- `load_tiles_tq3_1s` produces NaN after ~4 chunks
- q8_0 single-scale tile can't represent dual scales
- Fix options: rescale q values by d1/d0, or split into two half-blocks
- Not blocking — cuBLAS prefill is actually faster (221 vs 131 tok/s)

### 2. 9B Model Quality (Won't Fix)
- TQ3_1S at 3.5 bpw is too aggressive for 9B models (PPL ~19 vs Q4_0 ~8)
- Works well at 27B+ scale
- Not a kernel bug — verified CPU gives same bad PPL

### 3. Context Size Limited (Workaround Available)
- 27B TQ3_1S + fp16 KV: max ~2048 context on 16GB
- With TQ3_0 KV cache: 8192 context (working)
- For more: need second GPU or offload display to cheap card

## Server Configuration (Working)

```bash
# 8k context with TQ3_0 KV cache
llama-server \
    -m Qwen3.5-27B-TQ3_1S.gguf \
    -a qwen35-27b-tq3_1s \
    --host 0.0.0.0 --port 8085 \
    -ngl 99 -c 8192 -np 1 -ctk tq3_0 -ctv tq3_0 -fa on --no-warmup \
    --jinja --reasoning-budget 0 --reasoning-format deepseek
```

VRAM: 14.7 GB / 16.3 GB (1.5 GB free)
