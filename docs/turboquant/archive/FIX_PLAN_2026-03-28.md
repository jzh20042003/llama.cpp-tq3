### Tensor-Level Approximation Check

The next practical question was whether the adaptive block-level win could be approximated with today's
tensor-level override tooling.

On the same 9B witness, a whole-tensor greedy approximation was tested by promoting entire tensors to the
`18`-byte shared-shift variant under the same average bit budgets:

| Target | Per-block mix RMSE | Tensor-mix bpw | Tensor-mix RMSE | Tensor-mix dot |
|---|---:|---:|---:|---:|
| `4.05` | `0.002324` | `4.04` | `0.002390` | `0.013511` |
| `4.10` | `0.002305` | `4.08` | `0.002387` | `0.013490` |
| `4.20` | `0.002290` | `4.19` | `0.002378` | `0.013418` |

Additional signal at `4.05`:

- the adaptive policy promoted `333 / 3328` blocks
- those blocks cluster in a handful of tensors, but only as sparse hotspots inside those tensors
- the top tensor was `blk.15.attn_q.weight` with `60 / 256` promoted blocks
- within the top promoted tensors, the exceptions are still highly fragmented:
  - `blk.15.attn_q.weight`: `33` runs over `60` promoted blocks
  - `blk.15.attn_v.weight`: `43` runs over `49` promoted blocks
  - `blk.0.attn_qkv.weight`: `36` runs over `48` promoted blocks
  - top-12 summary: `257` runs over `331` promoted blocks

Interpretation:

- the gain is real, but it is not well approximated by promoting whole tensors
- it is also not naturally collapsing into a small number of long runs
- tensor-level overrides are too blunt for this weight line
- naive run-length exception sidecars are unlikely to be elegant enough on this witness
- if we want the adaptive `4.05-4.10 bpw` win on real models, we need:
  - a true mixed-block format
  - or another equally fine-grained surrogate

Current implication:

- do not treat tensor-level mixed GGUF as the main realization path for adaptive weights
- keep the adaptive result as proof that selective promotion has legs
- move the implementation focus toward block-level mixed formatting or an equivalent compact contract

Compact contract note:

- a flat per-block bitmap is cheap enough to keep this line viable
- for 32-weight blocks, a `1`-bit promotion flag costs only `0.03125 bpw`
- that means the practical realized budgets would still be:
  - `4.05` target -> about `4.081 bpw`
  - `4.10` target -> about `4.131 bpw`
  - `4.20` target -> about `4.231 bpw`

This is now the most realistic implementation direction for adaptive weights:

- `TQ3_1S` base blocks
- per-block bitmap for promoted exceptions
- compact side payload only for promoted blocks

Prototype status:

- the offline bitmap-sidecar representation now reconstructs the adaptive mixed result exactly
- on the 9B witness at the `4.05` target:
  - effective bpw: `4.08128`
  - RMSE: `0.002324`
  - dot RMSE: `0.013138`
  - reconstruction match vs ideal mixed output: `0.000000000`

This is the current best implementation lead on the weight side.

Working candidate format:

- name: `TQ3_1S-AP`
- base stream: `TQ3_1S` blocks for all blocks in the tensor
- selector: flat `1`-bit bitmap, `0 = base`, `1 = promoted`
- promoted payload: `18`-byte shared-shift blocks only for promoted blocks, stored in block order
- initial target band:
  - `TQ3_1S-AP-4.08`
  - `TQ3_1S-AP-4.13`

Why this is the right next runtime target:

- preserves the strong adaptive offline gain
- stays clearly below native `Q4_0` in effective bpw
- avoids the failure of tensor-level overrides
- keeps the promoted payload compact enough to preserve the `4.05-4.15 bpw` story
- keeps random access simple enough for `GET_ROWS` / dequant dispatch

Runtime constraint:

- `ggml` expects a fixed row size for a tensor
- the pure adaptive prototype has a variable number of promoted blocks per row
- so the first runtime design must choose one of:
  - a true mixed-block type with its own row contract
  - or auxiliary sidecar tensors (bitmap + promoted payload) attached to the base tensor

Fixed-quota-per-row fallback was tested as a possible row-regular surrogate on the 9B witness at the
`4.05` target:

|---|---:|---:|---:|
| `2` | `4.03816` | `0.002386` | `0.013469` |
| `4` | `4.04387` | `0.002377` | `0.013423` |
| `8` | `4.05183` | `0.002365` | `0.013347` |
| `12` | `4.05814` | `0.002356` | `0.013299` |
| `16` | `4.06385` | `0.002349` | `0.013252` |

Reference:

- full bitmap-sidecar adaptive result at the same target:
  - `4.08128 bpw`
  - `RMSE 0.002324`
  - `dot 0.013138`

Interpretation:

- a row-regular quota format is workable, but clearly weaker than the full bitmap-sidecar path
- it should be treated as a fallback implementation option, not the primary target
- the main runtime direction should stay:
  - general sidecar tensors
  - or a purpose-built mixed-block type

Auxiliary sidecar tensors look feasible in the current loader:

- `llama_model_loader` indexes arbitrary GGUF tensor names in `weights_map`
- optional extra tensors can be fetched by exact string name via:
  - `get_weight(name)`
  - `get_tensor_meta(name)`
- this means a first runtime realization does not need to force everything through the existing tensor-name enum

Current preferred runtime realization order:

1. sidecar tensors by name
   - `<tensor>.tq3_ap_bitmap`
   - `<tensor>.tq3_ap_promoted`
2. only if that proves too awkward, fall back to a dedicated mixed-block ggml type

Promoted-payload choice:

- on the 9B witness at the same `4.05` target:
  - shared-shift bitmap mix:
    - `RMSE 0.002324`
    - `dot 0.013138`
    - effective `4.08128 bpw`
  - `Q4_0` promoted sidecar gave stronger reconstruction quality offline,
    but it changed the storage economics completely because promoted blocks keep
    the base `TQ3_1S` bytes and add a full `Q4_0` block on top

Interpretation:

- `Q4_0` promoted sidecars are not product-viable at the intended `4.05-4.15 bpw` band
- they are useful as an upper-bound quality probe, not as the active format
- the active adaptive-weight line should stay:
  - `TQ3_1S` base
  - bitmap sidecar
  - `18`-byte shared-shift promoted payload

Additive fixed-row fallback status:

- a full promoted replacement block in a sidecar is too expensive because it keeps the base block
  and adds the full promoted block on top
- the only additive refinement with real legs so far is a mean-only sidecar on top of `TQ3_1S`

## External Research Review

This section captures the useful parts of newer external ideas and how they map to the current plan.

### 1. `K` / `V` Asymmetry Is Strongly Supported

The best-supported external result is that keys and values should not be treated symmetrically.

Useful sources:

- `KVLinC`
- `More For Keys, Less For Values`

Practical interpretation:

- keys are the fragile side
- values are the safer compression target
- this matches our own best current KV policy candidate:
  - `K=q8_0`
  - `V=tq3_0`

Current action implication:

- keep the KV line focused on improving compressed `V`
- do not spend near-term effort trying to push low-bit `K` harder than `V`

### 2. WUSH Is A Real Lead, But Not A Free Drop-In

The WUSH result is directionally important:

- fixed random Hadamard is not the best possible transform
- adaptive or data-aware transforms can lower quantization error

But for this project, it should be treated as:

- a research lead
- not an immediate “free quality win”

Why:

- a model-specific or block-aware transform can increase runtime complexity
- OpenVINO also describes fused or online Hadamard ideas in a context where implementation details matter
- we should not assume “better transform = same speed” without a prototype

Current action implication:

- keep WUSH-style transforms as an offline prototype target
- do not replace the active execution track with it yet

### 3. Weight-Line Lesson: The Remaining Gap Is Not A Codebook Problem

Our own evidence now matters more than speculation here.

What was tested:

- global `TQ3_1S` codebook retune on 9B
- family-specific codebooks on 9B
- mixed `ffn_down -> Q4_0`

What happened:

- global codebook retune worsened 9B model-level PPL
- family-specific codebooks worsened the offline prototype metrics
- `ffn_down` override did not matter on 9B

Conclusion:

- the remaining `TQ3_1S` weight gap is not primarily a centroid/codebook issue
- the next weight-quality lever is still better scale modeling / half-block calibration

### 4. KV-Line Lesson: `V`-First Is The Best Near-Term Bet

Because the external asymmetry result and our own measurements agree, the KV roadmap should stay:

1. preserve or improve `K=q8_0`
2. keep compressing `V`
3. improve the compressed `V` path before revisiting more aggressive `K`

This is the highest-confidence route to:

- better long-context fit
- fewer quality regressions
- cleaner justification for future public claims

## Consolidated Priority Ranking

Current best ROI:

1. `TQ3_1S` scale-model improvement for weights
2. `TQ3_0` adaptive-KV with `K=q8_0, V=tq3_0`
3. `V`-focused KV math improvements
